In the tradition of Legend of the Red Dragon (1992)

What Sorcery Is This?

In 1991, Seth Able created Legend of the Red Dragon — a text-based RPG that ran on BBS servers, where knights fought monsters, traded at shops, and battled each other through a 2400-baud modem. The game needed a trusted host: a sysop running the server, managing the database, enforcing the rules.

⛓

The Old Way

BBS door games required a central server. The sysop was the consensus layer. They stored your character, validated your moves, and could wipe the database on a whim. Trust was social. Cheating was a whisper away.

◈

The BlockDAG Way

Every character sheet is a covenant-enforced UTXO. Every sword swing is a state transition validated by ZK proof. Every trade is an atomic ICC transaction between Player and Shop covenants. No server. No sysop. No trust required.

✦

Why It Matters

This isn't nostalgia — it's a proof of concept. If Kaspa's covenant stack can run a multiplayer RPG with persistent state, daily resets, PvP combat, and an economy — it can run anything. DeFi, DAOs, games, and applications we haven't imagined yet.

How The Covenants Map

Character Sheet → Self-preserving UTXO with validateOutputState()
Forest Combat → Episodic Game covenant + RISC Zero proof of valid moves
Shop Purchase → ICC atomic tx: Player covenant + Shop covenant co-validate
PvP Arena → Two Player covenants + Arena covenant, timeout via this.age
Daily Fight Limit → UTXO age introspection: this.age > 86400 resets counter
Gold & Stakes → Native KAS locked in covenant, enforced via KIP-10 introspection

Status: Three Covenants + ICC + Verifiable Combat LIVE on TN12

Character creation deploys three covenants in one transaction: Player (160 bytes), Shop (56 bytes), and Opponent (80 bytes) — 296 bytes total on Kaspa Testnet-12. Every game action updates state on-chain via validateOutputState, owner-signed with Schnorr. Shop purchases and PvP combat use Inter-Covenant Communication: two covenants updated atomically in one tx. Combat is deterministic — seeded from the BlockDAG's tip hash, replayable and verifiable by anyone. No backend. Browser talks to our TN12 node via wRPC.
